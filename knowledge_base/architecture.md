# 架构设计原则

## 分层架构

### 经典三层/四层架构
- 表现层（Controller/Handler）：接收请求、参数校验、返回响应
- 业务层（Service）：核心业务逻辑、事务管理
- 数据层（Repository/DAO）：数据库访问、数据持久化
- 领域层（Domain/Model）：业务实体、值对象、领域规则
- 每层只依赖下一层，不能跨层调用

### 层间通信
- 上层通过接口调用下层，不直接依赖实现
- DTO（Data Transfer Object）在层间传递数据，不要直接暴露数据库实体
- 不同层的异常类型不同，需要转换

## 微服务 vs 单体

### 单体架构
- 优点：简单、部署方便、调试容易、事务简单
- 缺点：代码耦合、部署粒度大、技术栈单一、团队协作冲突
- 适合：小团队、早期项目、业务简单

### 微服务架构
- 优点：独立部署、技术栈灵活、团队自治、故障隔离
- 缺点：分布式复杂性、网络延迟、数据一致性、运维成本高
- 适合：大团队、业务复杂、需要独立扩展的场景
- 不要为了微服务而微服务，先单体后拆分

### 拆分原则
- 按业务领域拆分（DDD 限界上下文），不要按技术层拆分
- 每个服务有自己的数据库，不共享数据库
- 服务间通过 API 或消息队列通信
- 拆分粒度：一个服务由一个小团队（2-pizza team）维护

## API 设计

### RESTful 设计
- 资源用名词复数：/users, /orders, /products
- HTTP 方法表达操作：GET 查询、POST 创建、PUT 全量更新、PATCH 部分更新、DELETE 删除
- 嵌套资源：/users/{id}/orders
- 过滤排序分页用查询参数：?status=active&sort=created_at&page=1&size=20
- 版本控制：/api/v1/users

### 响应设计
- 统一响应格式：{ code, message, data }
- 成功：200（查询）、201（创建）、204（删除）
- 客户端错误：400（参数错误）、401（未认证）、403（无权限）、404（不存在）、409（冲突）
- 服务端错误：500（内部错误）、502（网关错误）、503（服务不可用）
- 错误响应包含错误码和可读描述

### GraphQL
- 适合前端需要灵活查询的场景，避免 over-fetching 和 under-fetching
- 强类型 Schema 定义，自带文档
- N+1 问题用 DataLoader 解决
- 不适合简单 CRUD，引入复杂度不值得

## 消息队列

- 场景：异步处理、削峰填谷、系统解耦、事件驱动
- 选型：Kafka（高吞吐、日志流）、RabbitMQ（灵活路由）、Redis Stream（轻量）
- 消息可靠性：至少一次投递 + 消费端幂等
- 死信队列处理消费失败的消息
- 消息顺序性：同一分区/队列内有序

## 数据一致性

- 强一致性：分布式事务（2PC），性能差但数据准确
- 最终一致性：事件驱动 + 补偿机制，性能好但有延迟
- Saga 模式：长事务拆分为多个本地事务 + 补偿操作
- 幂等性设计：重复操作不产生副作用（唯一键、状态机）
- CAP 定理：一致性、可用性、分区容错性只能三选二

## 可观测性

- 日志（Logging）：记录事件和错误，结构化 JSON 格式
- 指标（Metrics）：量化系统状态，如 QPS、延迟、错误率
- 链路追踪（Tracing）：跟踪请求在多个服务间的调用链
- 三者结合才能全面了解系统状态
- 告警规则：基于指标设置阈值，异常时自动通知
