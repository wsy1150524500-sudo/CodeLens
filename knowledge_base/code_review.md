# Code Review Checklist

## 正确性

- [ ] 逻辑是否正确，边界条件是否处理（空值、零值、负数、空集合）
- [ ] 错误处理是否完善，异常是否被正确捕获和传播
- [ ] 并发场景是否有竞态条件（race condition）
- [ ] 资源是否正确释放（文件句柄、数据库连接、锁）
- [ ] 类型转换是否安全，是否有溢出风险

## 安全性

- [ ] 用户输入是否经过校验和转义
- [ ] SQL 是否使用参数化查询
- [ ] 敏感信息是否脱敏（日志、错误信息、API 响应）
- [ ] 认证和授权是否正确实现
- [ ] 依赖是否有已知漏洞

## 性能

- [ ] 是否有不必要的数据库查询（N+1 问题）
- [ ] 循环内是否有可以提到循环外的计算
- [ ] 大数据集是否分页或流式处理
- [ ] 是否有适合缓存的重复计算
- [ ] 算法复杂度是否合理

## 可读性

- [ ] 命名是否清晰表达意图
- [ ] 函数是否足够短小，职责是否单一
- [ ] 嵌套层级是否过深（建议不超过 3 层）
- [ ] 复杂逻辑是否有必要的注释
- [ ] 代码格式是否一致（缩进、空行、括号风格）

## 可维护性

- [ ] 是否有重复代码可以提取
- [ ] 是否有硬编码的魔法数字或字符串
- [ ] 模块间耦合度是否合理
- [ ] 是否易于测试（依赖是否可注入）
- [ ] 是否符合项目现有的架构和约定

## 测试

- [ ] 新功能是否有对应的单元测试
- [ ] 边界条件和异常路径是否有测试覆盖
- [ ] 测试是否独立，不依赖执行顺序或外部状态
- [ ] 测试命名是否描述了被测试的行为
- [ ] mock 是否合理，没有过度 mock 导致测试失去意义

## Review 沟通原则

- 对事不对人：评论代码，不评价作者
- 提出问题而非命令："这里是否考虑过并发场景？" 优于 "这里有 bug"
- 给出理由和替代方案，不要只说"不好"
- 区分必须修改（blocker）和建议优化（nit）
- 及时响应，不要让 PR 挂太久
