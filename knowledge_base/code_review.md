# Code Review Checklist

## 正确性

- 逻辑是否正确，边界条件是否处理（空值、零值、负数、空集合、最大值）
- 错误处理是否完善，异常是否被正确捕获和传播
- 并发场景是否有竞态条件（race condition）
- 资源是否正确释放（文件句柄、数据库连接、锁、网络连接）
- 类型转换是否安全，是否有溢出风险
- 浮点数比较是否使用了容差（epsilon），而非直接 ==
- 时区处理是否正确，日期计算是否考虑了闰年、夏令时
- 字符编码是否统一（UTF-8），是否处理了多字节字符
- 递归是否有终止条件，是否可能栈溢出

## 安全性

- 用户输入是否经过校验和转义
- SQL 是否使用参数化查询
- 敏感信息是否脱敏（日志、错误信息、API 响应）
- 认证和授权是否正确实现
- 依赖是否有已知漏洞
- 文件操作是否防止了路径遍历攻击
- 加密算法是否使用了现代标准（AES-256, RSA-2048+）
- 随机数生成是否使用了密码学安全的随机源

## 性能

- 是否有不必要的数据库查询（N+1 问题）
- 循环内是否有可以提到循环外的计算
- 大数据集是否分页或流式处理
- 是否有适合缓存的重复计算
- 算法复杂度是否合理
- 是否有不必要的内存分配（循环内创建对象）
- 数据库查询是否走了索引
- 是否有可以并行化的独立任务

## 可读性

- 命名是否清晰表达意图
- 函数是否足够短小，职责是否单一
- 嵌套层级是否过深（建议不超过 3 层）
- 复杂逻辑是否有必要的注释
- 代码格式是否一致（缩进、空行、括号风格）
- 是否有"聪明"但难以理解的写法，能否用更直白的方式
- 控制流是否清晰，有没有 goto 或多层 break/continue

## 可维护性

- 是否有重复代码可以提取
- 是否有硬编码的魔法数字或字符串
- 模块间耦合度是否合理
- 是否易于测试（依赖是否可注入）
- 是否符合项目现有的架构和约定
- 配置是否外部化，而非写死在代码中
- 是否有适当的抽象层，方便替换实现

## 测试

- 新功能是否有对应的单元测试
- 边界条件和异常路径是否有测试覆盖
- 测试是否独立，不依赖执行顺序或外部状态
- 测试命名是否描述了被测试的行为
- mock 是否合理，没有过度 mock 导致测试失去意义
- 是否有集成测试覆盖关键路径
- 测试数据是否有代表性，覆盖了正常和异常场景
- 测试是否可重复执行，不依赖外部服务状态

## API 设计

- RESTful 接口是否语义正确（GET 读取、POST 创建、PUT 更新、DELETE 删除）
- 响应状态码是否准确（200/201/400/401/403/404/500）
- 错误响应是否包含有用的错误信息和错误码
- 分页接口是否返回总数和分页信息
- API 版本控制（URL 路径 /v1/ 或 Header）
- 请求/响应格式是否一致（统一用 JSON，字段命名风格统一）
- 是否有接口文档（OpenAPI/Swagger）
- 幂等性：重复调用是否产生相同结果

## 数据库变更

- 新增字段是否有默认值，是否允许 NULL
- 索引是否合理，是否会影响写入性能
- 数据迁移脚本是否可回滚
- 大表变更是否考虑了锁表风险
- 外键约束是否必要，是否影响性能
- 字段类型是否合适（varchar 长度、数值精度）

## Review 沟通原则

- 对事不对人：评论代码，不评价作者
- 提出问题而非命令："这里是否考虑过并发场景？" 优于 "这里有 bug"
- 给出理由和替代方案，不要只说"不好"
- 区分必须修改（blocker）和建议优化（nit）
- 及时响应，不要让 PR 挂太久
- 肯定好的代码，不要只挑毛病
- 大的架构问题线下讨论，不要在 PR 里争论
