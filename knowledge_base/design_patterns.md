# 设计模式速查

## 创建型模式

### 单例模式（Singleton）
- 场景：全局唯一实例，如配置管理、连接池、日志器
- 要点：线程安全的懒加载，避免全局状态滥用
- 警告：过度使用单例会导致隐式耦合和测试困难
- Python 实现：模块级变量天然是单例；或用 __new__ 控制实例化
- Java 实现：双重检查锁定（DCL）或枚举单例
- Go 实现：sync.Once 保证只初始化一次

### 工厂模式（Factory）
- 场景：根据条件创建不同类型的对象，解耦创建逻辑
- 简单工厂：一个方法根据参数返回不同实例
- 工厂方法：定义创建接口，子类决定实例化哪个类
- 抽象工厂：创建一族相关对象，如跨平台 UI 组件
- 优先用工厂方法替代复杂的 if-else 创建逻辑
- 配合依赖注入使用效果更好

### 建造者模式（Builder）
- 场景：构造参数多且部分可选的复杂对象
- 链式调用提升可读性：Query.select("name").where("age > 18").limit(10)
- 替代多参数构造函数和 telescoping constructor 反模式
- Java 中 Lombok @Builder 自动生成
- Python 中 dataclass + 默认值可以部分替代

### 原型模式（Prototype）
- 场景：创建对象成本高（如深拷贝复杂对象），通过克隆已有实例
- 注意深拷贝和浅拷贝的区别
- Python: copy.deepcopy()，Java: Cloneable 接口
- 适用于配置模板、游戏对象等需要大量相似实例的场景

## 结构型模式

### 适配器模式（Adapter）
- 场景：对接第三方库或旧系统，接口不兼容时做转换层
- 包装旧接口，暴露新接口，不修改原有代码
- 类适配器（继承）vs 对象适配器（组合），优先用组合
- 实际应用：数据库驱动适配、日志框架适配（SLF4J）

### 装饰器模式（Decorator）
- 场景：动态给对象添加职责，如日志、缓存、权限检查、重试
- Python 的 @decorator、Java 的 I/O Stream 都是典型应用
- 优于继承：可以组合多个装饰器，避免类爆炸
- 注意装饰器顺序会影响行为
- TypeScript/JavaScript 的装饰器语法（@log, @cache）

### 代理模式（Proxy）
- 场景：延迟加载、访问控制、远程调用、缓存代理
- 与装饰器的区别：代理控制访问，装饰器增强功能
- 动态代理：Java Proxy、Python __getattr__
- 实际应用：ORM 的懒加载、RPC 框架的客户端代理

### 外观模式（Facade）
- 场景：为复杂子系统提供简化接口
- 不是封装，是提供一个更易用的入口
- 实际应用：SDK 封装、API Gateway、工具类

### 组合模式（Composite）
- 场景：树形结构，如文件系统、组织架构、UI 组件树
- 统一处理叶子节点和容器节点
- 递归遍历树结构

## 行为型模式

### 策略模式（Strategy）
- 场景：同一行为有多种算法实现，运行时切换
- 用策略替代大段 if-else/switch：支付方式、排序算法、验证规则
- 结合工厂模式：根据配置或参数自动选择策略
- Python 中函数是一等公民，可以直接传函数作为策略
- Java 中用接口 + Lambda 表达式简化策略实现

### 观察者模式（Observer）
- 场景：一对多的事件通知，如 UI 事件、消息订阅、数据变更通知
- 发布-订阅是其变体，加了中间层（事件总线）解耦发布者和订阅者
- 注意：避免循环触发和内存泄漏（及时取消订阅）
- 实际应用：Vue 的响应式系统、React 的 setState、EventEmitter
- 分布式场景用消息队列（Kafka、RabbitMQ）实现

### 模板方法模式（Template Method）
- 场景：算法骨架固定，部分步骤由子类实现
- 典型应用：框架的生命周期钩子、数据处理 pipeline
- 用抽象方法定义可变步骤，基类控制流程
- Python 中用 abc.ABC 和 @abstractmethod
- 好莱坞原则："Don't call us, we'll call you"

### 责任链模式（Chain of Responsibility）
- 场景：请求需要经过多个处理器，如中间件、过滤器、审批流
- 每个处理器决定自己处理还是传递给下一个
- Web 框架的中间件机制就是典型的责任链
- 实际应用：Express/Koa 中间件、Java Servlet Filter、Django Middleware

### 命令模式（Command）
- 场景：将请求封装为对象，支持撤销/重做、队列、日志
- 实际应用：编辑器的 Undo/Redo、任务队列、宏录制
- 结合备忘录模式实现状态快照和回滚

### 迭代器模式（Iterator）
- 场景：统一遍历不同数据结构的方式
- Python 的 __iter__/__next__、Java 的 Iterator 接口
- 生成器（yield）是迭代器的语法糖，惰性求值节省内存
- for-of/for-in 循环底层都是迭代器协议

### 状态模式（State）
- 场景：对象行为随内部状态改变，如订单状态机、游戏角色状态
- 用状态对象替代大量的 if-else 状态判断
- 每个状态是一个类，封装该状态下的行为和转换规则

## 设计原则（SOLID）

- S 单一职责：一个类只有一个变化的原因
- O 开闭原则：对扩展开放，对修改关闭（通过抽象和多态实现）
- L 里氏替换：子类能替换父类而不破坏程序（子类不能削弱父类的契约）
- I 接口隔离：不要强迫实现不需要的接口方法（接口要小而专）
- D 依赖倒置：高层模块不依赖低层模块，都依赖抽象

## 其他重要原则

- KISS（Keep It Simple, Stupid）：保持简单，不要过度设计
- YAGNI（You Aren't Gonna Need It）：不要实现当前不需要的功能
- 组合优于继承：优先使用对象组合而非类继承
- 迪米特法则（最少知识原则）：一个对象应该对其他对象有最少的了解
- 关注点分离：不同的关注点放在不同的模块中
