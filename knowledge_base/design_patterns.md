# 设计模式速查

## 创建型模式

### 单例模式（Singleton）
- 场景：全局唯一实例，如配置管理、连接池、日志器
- 要点：线程安全的懒加载，避免全局状态滥用
- 警告：过度使用单例会导致隐式耦合和测试困难

### 工厂模式（Factory）
- 场景：根据条件创建不同类型的对象，解耦创建逻辑
- 简单工厂：一个方法根据参数返回不同实例
- 抽象工厂：创建一族相关对象，如跨平台 UI 组件
- 优先用工厂方法替代复杂的 if-else 创建逻辑

### 建造者模式（Builder）
- 场景：构造参数多且部分可选的复杂对象
- 链式调用提升可读性：Query.select("name").where("age > 18").limit(10)
- 替代多参数构造函数和 telescoping constructor 反模式

## 结构型模式

### 适配器模式（Adapter）
- 场景：对接第三方库或旧系统，接口不兼容时做转换层
- 包装旧接口，暴露新接口，不修改原有代码

### 装饰器模式（Decorator）
- 场景：动态给对象添加职责，如日志、缓存、权限检查、重试
- Python 的 @decorator、Java 的 I/O Stream 都是典型应用
- 优于继承：可以组合多个装饰器，避免类爆炸

### 代理模式（Proxy）
- 场景：延迟加载、访问控制、远程调用、缓存代理
- 与装饰器的区别：代理控制访问，装饰器增强功能

## 行为型模式

### 策略模式（Strategy）
- 场景：同一行为有多种算法实现，运行时切换
- 用策略替代大段 if-else/switch：支付方式、排序算法、验证规则
- 结合工厂模式：根据配置或参数自动选择策略

### 观察者模式（Observer）
- 场景：一对多的事件通知，如 UI 事件、消息订阅、数据变更通知
- 发布-订阅是其变体，加了中间层解耦发布者和订阅者
- 注意：避免循环触发和内存泄漏（及时取消订阅）

### 模板方法模式（Template Method）
- 场景：算法骨架固定，部分步骤由子类实现
- 典型应用：框架的生命周期钩子、数据处理 pipeline
- 用抽象方法定义可变步骤，基类控制流程

### 责任链模式（Chain of Responsibility）
- 场景：请求需要经过多个处理器，如中间件、过滤器、审批流
- 每个处理器决定自己处理还是传递给下一个
- Web 框架的中间件机制就是典型的责任链

## 设计原则（SOLID）

- S 单一职责：一个类只有一个变化的原因
- O 开闭原则：对扩展开放，对修改关闭
- L 里氏替换：子类能替换父类而不破坏程序
- I 接口隔离：不要强迫实现不需要的接口方法
- D 依赖倒置：高层模块不依赖低层模块，都依赖抽象
