# 性能优化模式

## 算法与数据结构

- 理解时间复杂度：O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2^n)
- 频繁查找用哈希表（dict/HashMap/Map），不要遍历列表
- 频繁插入删除用链表，频繁随机访问用数组
- 排序后的数据用二分查找，不要线性扫描
- 避免在循环内做 O(n) 操作导致 O(n²)：循环内查找应预先建索引
- 字符串拼接在循环中用 StringBuilder/join，不要用 + 累加
- 选择合适的排序算法：小数据量用插入排序，大数据量用快排/归并
- 前缀和/差分数组优化区间查询
- 位运算替代简单的乘除和取模操作
- Trie 树优化前缀匹配场景

## 内存管理

- 大集合处理用迭代器/生成器（yield），不要一次性加载到内存
- 及时释放不再使用的大对象引用
- 注意闭包和事件监听器导致的内存泄漏
- 缓存要设上限和淘汰策略（LRU），避免无限增长
- 数据库查询只 SELECT 需要的字段，不要 SELECT *
- 分页加载大数据集，避免一次查询百万行
- 对象池复用频繁创建销毁的对象（数据库连接、线程）
- 弱引用（WeakRef/WeakMap）用于缓存场景，允许 GC 回收
- 避免在循环中创建对象，提到循环外复用
- 大文件处理用流式读取（stream），不要 readAll

## 并发与异步

- I/O 密集型任务用异步（async/await）或线程池
- CPU 密集型任务用多进程或 Worker 线程
- 数据库连接使用连接池，不要每次请求新建连接
- 避免锁竞争：缩小临界区范围，优先使用无锁数据结构
- 批量操作优于逐条操作：批量插入、批量请求
- 合理设置超时，避免无限等待阻塞线程
- 协程/goroutine 比线程更轻量，适合高并发 I/O 场景
- 信号量控制并发上限，防止资源耗尽
- 死锁预防：按固定顺序获取锁，设置锁超时
- 读写锁（RWLock）优化读多写少场景

## 缓存策略

- 读多写少的数据适合缓存：配置、热点查询结果、计算结果
- 缓存层级：本地缓存（进程内）→ 分布式缓存（Redis）→ CDN
- 缓存失效策略：TTL 过期、主动失效、版本号
- 防止缓存穿透：布隆过滤器或缓存空值
- 防止缓存雪崩：过期时间加随机偏移，热点数据永不过期
- 防止缓存击穿：互斥锁或 singleflight 保证只有一个请求回源
- HTTP 缓存：合理设置 Cache-Control、ETag、Last-Modified
- 缓存预热：系统启动时主动加载热点数据
- 多级缓存一致性：先更新数据库再删缓存（Cache Aside 模式）

## 数据库优化

- 慢查询先看执行计划（EXPLAIN），确认是否走索引
- 高频查询字段建索引，但索引不是越多越好（影响写入性能）
- 避免在 WHERE 条件中对字段做函数运算（会导致索引失效）
- N+1 查询问题：用 JOIN 或批量查询替代循环内单条查询
- 大表分页用游标分页（WHERE id > last_id）而非 OFFSET
- 读写分离：主库写、从库读
- 连接池大小 = CPU 核心数 * 2 + 磁盘数（经验公式）
- 批量写入用事务包裹，减少提交次数
- 避免长事务，锁持有时间越短越好
- 冷热数据分离：历史数据归档到单独的表或库

## 前端性能

- 首屏加载优化：代码分割（lazy import）、Tree Shaking、压缩
- 图片优化：WebP 格式、懒加载、响应式图片（srcset）
- 虚拟滚动处理长列表（vue-virtual-scroller、react-window）
- 防抖（debounce）和节流（throttle）控制高频事件
- Web Worker 处理 CPU 密集计算，不阻塞主线程
- 减少 DOM 操作，批量更新用 requestAnimationFrame
- 预加载关键资源：preload、prefetch、preconnect
- Service Worker 缓存静态资源，支持离线访问

## 网络优化

- 减少 HTTP 请求数：合并请求、雪碧图、内联小资源
- 启用 Gzip/Brotli 压缩
- HTTP/2 多路复用，减少连接开销
- API 响应分页，不要一次返回全部数据
- 长轮询/WebSocket/SSE 替代频繁轮询
- CDN 加速静态资源分发
- DNS 预解析：dns-prefetch
- 接口合并：BFF 层聚合多个后端接口
