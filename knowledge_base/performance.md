# 性能优化模式

## 算法与数据结构

- 理解时间复杂度：O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2^n)
- 频繁查找用哈希表（dict/HashMap/Map），不要遍历列表
- 频繁插入删除用链表，频繁随机访问用数组
- 排序后的数据用二分查找，不要线性扫描
- 避免在循环内做 O(n) 操作导致 O(n²)：循环内查找应预先建索引
- 字符串拼接在循环中用 StringBuilder/join，不要用 + 累加

## 内存管理

- 大集合处理用迭代器/生成器（yield），不要一次性加载到内存
- 及时释放不再使用的大对象引用
- 注意闭包和事件监听器导致的内存泄漏
- 缓存要设上限和淘汰策略（LRU），避免无限增长
- 数据库查询只 SELECT 需要的字段，不要 SELECT *
- 分页加载大数据集，避免一次查询百万行

## 并发与异步

- I/O 密集型任务用异步（async/await）或线程池
- CPU 密集型任务用多进程或 Worker 线程
- 数据库连接使用连接池，不要每次请求新建连接
- 避免锁竞争：缩小临界区范围，优先使用无锁数据结构
- 批量操作优于逐条操作：批量插入、批量请求
- 合理设置超时，避免无限等待阻塞线程

## 缓存策略

- 读多写少的数据适合缓存：配置、热点查询结果、计算结果
- 缓存层级：本地缓存（进程内）→ 分布式缓存（Redis）→ CDN
- 缓存失效策略：TTL 过期、主动失效、版本号
- 防止缓存穿透：布隆过滤器或缓存空值
- 防止缓存雪崩：过期时间加随机偏移，热点数据永不过期

## 数据库优化

- 慢查询先看执行计划（EXPLAIN），确认是否走索引
- 高频查询字段建索引，但索引不是越多越好（影响写入性能）
- 避免在 WHERE 条件中对字段做函数运算（会导致索引失效）
- N+1 查询问题：用 JOIN 或批量查询替代循环内单条查询
- 大表分页用游标分页（WHERE id > last_id）而非 OFFSET
- 读写分离：主库写、从库读
