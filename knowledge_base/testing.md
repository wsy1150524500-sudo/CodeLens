# 测试最佳实践

## 测试金字塔

- 单元测试（底层，数量最多）：测试单个函数/方法，快速、隔离、稳定
- 集成测试（中层）：测试模块间交互，如 API + 数据库、服务间调用
- 端到端测试（顶层，数量最少）：模拟真实用户操作，覆盖完整流程
- 比例建议：70% 单元 / 20% 集成 / 10% E2E

## 单元测试原则

### FIRST 原则
- Fast：毫秒级执行，开发者愿意频繁运行
- Isolated：不依赖外部服务、文件系统、数据库
- Repeatable：任何环境下运行结果一致
- Self-validating：自动判断通过/失败，不需要人工检查
- Timely：和生产代码同时编写，不要事后补

### AAA 模式
- Arrange：准备测试数据和环境
- Act：执行被测试的操作
- Assert：验证结果是否符合预期
- 每个测试只测一个行为，一个 Assert 主题

### 测试命名
- 描述被测试的行为和预期结果
- 格式：test_[被测方法]_[场景]_[预期结果]
- 示例：test_withdraw_insufficient_balance_raises_error
- 好的测试名就是文档

## Mock 与 Stub

- Stub：返回预设值，不验证调用（替代外部依赖的返回值）
- Mock：验证交互行为（是否调用了某方法、调用了几次、参数是什么）
- 不要 mock 你不拥有的代码，用适配器包装后 mock 适配器
- 过度 mock 会导致测试和实现强耦合，重构时测试全挂
- 优先用真实对象，只在必要时 mock（外部 API、数据库、时间）

## 测试覆盖率

- 行覆盖率 80% 是合理目标，100% 不现实也不必要
- 关注分支覆盖率，比行覆盖率更有意义
- 高覆盖率不等于高质量测试，要关注断言的有效性
- 核心业务逻辑和边界条件必须覆盖
- 不要为了覆盖率而写无意义的测试

## 测试反模式

- 脆弱测试：依赖实现细节，代码一改测试就挂
- 慢测试：依赖网络、数据库、文件系统，拖慢 CI
- 互相依赖的测试：测试 A 的结果影响测试 B
- 过度 mock：mock 了所有东西，测试没有实际意义
- 测试中的逻辑：测试代码里有 if/for/try，测试本身可能有 bug
- 魔法数字：测试数据没有解释为什么用这个值

## TDD（测试驱动开发）

- Red：先写一个失败的测试
- Green：写最少的代码让测试通过
- Refactor：重构代码，保持测试通过
- 适合逻辑复杂、边界条件多的场景
- 不适合 UI 开发、探索性编程
